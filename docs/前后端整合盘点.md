# 前后端整合盘点

> 盘点日期：2026-07-12  
> 当前阶段：仅记录，不修改前后端业务代码，不删除 Mock。

## 1. 结论

前端目前注册 6 个业务页面：`/overview`、`/demands`、`/demands/:id`、`/realtime`、`/strategies`、`/revenue`（`frontend/src/App.tsx:17`）。

后端采用 Flask，业务接口统一位于 `/api/v1`。核心需求、评估、策略、资源和供应商接口大多已经存在，但前端 API 门面仍主要导出 `mockApi`（`frontend/src/api/kongming.ts:5`）。当前只有资源看板、资源修改、资源集群列表和供应商配额请求真实接口，且资源相关请求失败后会静默回退 Mock（`frontend/src/api/kongming.ts:7`、`:25`、`:48`）。

整合工作可分为三类：

1. **已有接口待接入**：需求、评估、策略、报告和大部分运营数据。
2. **字段或语义需适配**：分页、枚举、嵌套对象、资源修改响应、策略确认语义等。
3. **后端接口缺失**：策略大屏专用聚合、部分收益聚合、前端按钮对应的批量或状态操作。

## 2. 当前 Mock 与本地数据来源

### 2.1 集中 Mock

- `frontend/src/api/mockData.ts`：保存业务实体、图表序列、资源、收益、报告等主要模拟数据，并实现内存 CRUD。
- `frontend/src/api/kongming.ts:2`：导入 `mockApi`。
- `frontend/src/api/kongming.ts:5`：`dashboardsApi` 通过展开 `mockApi.dashboards` 暴露模拟接口。
- `frontend/src/api/kongming.ts:62`：需求、评估、策略、报告等 API 直接指向 `mockApi`。
- `frontend/src/api/kongming.ts:10`、`:28`、`:51`：真实资源请求失败后静默返回 Mock，可能掩盖 404、字段错误和后端异常。

### 2.2 页面本地数据

- `OperationsDashboard`：运营指标、报告内容及部分列表来自 Mock；资源概览尝试真实接口。
- `DemandListPage`：列表、筛选、分页和导出数据来自 Mock API。
- `DemandDetailPage`：需求详情、评估、策略及确认/放弃等操作来自 Mock API；部分展示字段由页面拼装。
- `RealtimeDashboard`：资源和供应商配额已请求真实接口，但资源存在静默 Mock fallback；页面仍有本地筛选、轮询和展示派生数据。
- `StrategyDashboard`：大量策略卡片、收益数字、迁移流向及交互状态由页面内数组/对象或 Mock API 提供，是当前硬编码最集中的页面。
- `RevenueDashboardPage`：收益摘要、趋势、客户/模型排行、分成与结算展示来自 Mock API 或前端派生。

### 2.3 未挂载页面

`frontend/src/pages` 仍存在未注册到 `App.tsx` 的页面。这些页面也包含 Mock/硬编码数据，但不属于当前 6 个可访问业务路由。删除 `mockData.ts` 前必须确认这些页面是后续恢复路由还是删除，否则会产生残留引用。

## 3. 逐页接口映射

### 3.1 运营总览 `/overview`

前端需求：

- 运营摘要：待处理需求、待评估数、草稿策略数、近 24 小时收益、告警数。
- 资源摘要与集群列表。
- 需求和策略简表。
- 周报、月报生成或查询。

后端现状：

- 资源摘要：`GET /api/v1/dashboard/resources`，已局部接入。
- 需求列表：`GET /api/v1/demands`，已有未接入。
- 策略列表：`GET /api/v1/policies`，已有未接入。
- 报告接口：后端已有 reports 域接口，前端仍走 Mock。

缺口或风险：

- 运营摘要中的 `revenue_last_24h`、`open_alerts` 等是否由单个 dashboard 接口完整返回，需要以实际响应确认。
- 若后端仅提供明细，前端不应自行拉取多页后聚合；建议补充稳定的运营摘要接口。
- 周报/月报需要确认是同步返回内容、异步任务，还是查询已生成报告；当前 Mock 语义不足以确定。

### 3.2 需求列表 `/demands`

前端需求：分页、状态/模型/客户筛选、排序、详情跳转、导出。

后端现状：

- `GET /api/v1/demands` 已支持分页和查询参数。
- 需求创建、更新、状态操作等核心接口已存在。

字段适配：

- 前端统一分页类型为 `{items,page,page_size,total}`（`frontend/src/api/types.ts:8`），需验证后端 envelope 解包后的分页结构完全一致。
- 前端 `Demand` 期望 `report_id`、`customer_id`、`model_name`、`expected_tpm`、`expected_rpm` 等字段（`frontend/src/api/types.ts:21`）；需要逐字段核对数据库空值和数值序列化。
- 筛选枚举及状态文案必须以后端枚举为准，不能保留 Mock 中自定义状态。

可能缺失：

- 若“导出”要求下载文件，当前列表 JSON 接口不足，需要专用导出接口或明确由前端导出当前查询结果。
- 若页面支持批量状态操作，需要确认后端是否有批量接口。

### 3.3 需求详情 `/demands/:id`

前端需求：需求详情、关联评估、候选方案、策略详情、确认策略、放弃策略、重新评估。

后端现状：

- `GET /api/v1/demands/:id` 已有。
- evaluations 域已有列表、详情和求解相关接口。
- policies 域已有列表、详情、确认/应用一类接口。

字段或语义风险：

- 前端可能按需求 ID 直接获取单个评估对象，后端可能返回评估列表；需要明确“一次需求对应一个当前评估”还是保留历史记录。
- 页面使用的候选方案、资源分配、收益明细可能是嵌套结构，而后端部分字段来自 JSON 列；需要建立显式 TypeScript DTO，不能直接复用数据库字典。
- “人工确认”“应用”“接受”在前后端命名和状态流转上可能不是同一动作，应以后端状态机为准。
- 重新评估是同步求解还是创建异步任务需要确认，页面必须有对应 loading、失败和轮询状态。

可能缺失：

- 放弃候选方案或策略的专用接口。
- 按需求返回“当前评估 + 候选方案 + 当前策略”的详情聚合接口。现有接口可分别请求，但需要处理一致性和请求竞态。

### 3.4 实时资源 `/realtime`

前端需求：资源摘要、集群筛选、集群修改、供应商配额、刷新/轮询。

后端现状：

- `GET /api/v1/dashboard/resources` 已接入。
- 资源集群列表和修改接口已接入。
- 供应商配额接口已接入。

字段适配与风险：

- 删除所有资源请求的 `catch -> mockApi`，真实错误应进入页面错误态。
- `updateClusterResource` 当前前端把返回值声明为 `ResourceCluster`，但后端更新响应可能是操作结果或更新后的实体，需要实测确认。
- GPU 型号、机房、集群名和部署模型筛选参数的空值规则需要统一。
- 配额数值单位、更新时间、可用量与总量字段需要核对，避免百分比重复计算或单位混用。

缺失检查：

- 若页面按钮包含新增/删除集群、扩缩容或批量修改，需要确认对应写接口；当前已确认的是修改能力。

### 3.5 策略管理 `/strategies`

前端需求：按阶段展示需求评估、模型拟合、调度、闲时策略、忙时策略；展示模型迁移流向、机器数、收益；支持生成、查看、人工确认、修改、放弃。

后端现状：

- policies 和 evaluations 核心 CRUD/求解接口存在。
- 现有接口更偏实体和单次求解，不一定直接提供该大屏需要的聚合结构。

P0 缺口：

- 缺少与策略大屏五个模块一一对应的稳定聚合 DTO，前端当前大量依赖页面内硬编码结构。
- “闲时/忙时策略”的流向行需要源模型、目标模型、迁移机器数、单价/容量、预计收益、策略状态等字段；需要确认后端算法结果是否全部持久化并可查询。
- “生成”“修改”“放弃”按钮需要明确接口、请求体和状态流转。不能继续用本地数组模拟成功。

建议：

- 优先评估复用现有 evaluations/policies 响应，通过前端适配器组合；只有无法保证一致性或请求数过多时，再增加 `GET /api/v1/dashboard/strategies` 一类聚合接口。
- 后端为策略流向定义显式 schema，避免前端解析算法内部 JSON。

### 3.6 收益分析 `/revenue`

前端需求：收益摘要、趋势序列、客户排行、模型排行、分成/结算明细及筛选。

后端现状：

- 后端存在 revenue 域接口，可覆盖部分摘要、时间序列或明细查询。
- 前端仍主要使用 Mock 数据。

字段适配与缺口：

- 统一金额单位、币种、时间粒度、时区和小数精度。
- 趋势点字段需统一为明确的 `timestamp/date + value`，避免页面同时兼容多套字段。
- 客户和模型排行需要确认后端是否直接返回排名、占比及环比；若只有明细，不建议前端全量聚合。
- 分成、结算状态、账期和应付金额若后端没有稳定接口，需要新增。
- 页面筛选维度必须与后端 query 参数一致，尤其是日期范围、客户、模型和粒度。

## 4. 跨域契约问题

### P0：整合前必须解决

1. **移除 API 门面的 Mock 导出**：`kongming.ts` 不能继续通过 `...mockApi` 或直接别名导出业务 API。
2. **移除静默回退**：真实请求失败必须暴露错误，不得返回看似正常的 Mock 数据。
3. **建立显式 DTO**：为后端 envelope、分页、需求、评估、策略流向、资源和收益定义与实际响应一致的类型。
4. **明确状态机**：需求、评估和策略的状态值及“确认/应用/放弃/重新生成”动作必须以后端为唯一事实来源。
5. **补齐策略大屏数据契约**：这是当前硬编码最多、与后端实体接口差距最大的页面。

### P1：字段与行为适配

- 响应统一使用 `{data,message,request_id,errors}`，确认所有错误状态是否仍使用同一 envelope。
- 统一 ID 类型；前端基础实体当前使用数字 ID（`frontend/src/api/types.ts:15`），业务编号如 `report_id`、策略编号不得与数据库 ID 混用。
- 统一时间格式为带时区的 ISO 8601，并明确收益按哪个时区聚合。
- 统一可空字段。后端数据库允许 `NULL` 的字段在 TypeScript 中必须显式声明 `null`。
- 统一枚举，不使用自由字符串或仅用于 Mock 的状态。
- 写接口确认是返回更新后实体、任务信息还是仅操作结果，前端类型不得猜测。
- 对求解、报告生成等耗时操作明确同步/异步语义及轮询接口。

### P2：体验和维护性

- 为每个页面补充 loading、empty、error、retry 状态，替代 Mock 永远成功的假设。
- 明确列表导出由后端生成还是前端导出。
- 删除未挂载页面的 Mock 前先决定页面去留。
- 删除 `mockData.ts` 后用静态搜索确认无 `mockApi`、模拟延时、随机数据和硬编码业务实体残留。

## 5.2 待用户确认后再改（字段语义不明，本轮保留不动）

- `GET /revenue/dashboard`（收益看板 idle/busy/peak_shaving）：已确认“从已采纳 time_period 策略 summary_json 聚合推导”，但以下字段口径在现有算法产物中无明确定义，需确认后再实现：
  - idle 与 busy 的划分规则：time_period 求解是整段积分，`summary_json` 无 idle/busy 分段，一条策略如何拆成两组明细未定义。

## 6. 下一阶段验收标准

- `frontend/src` 中不存在 `mockApi`、`mockData` 或真实请求失败回退模拟数据的逻辑。
- 6 个已注册页面的业务数据全部来自后端或明确的纯展示常量。
- 所有前端写操作均有对应后端接口和失败反馈。
- TypeScript DTO 与实际响应样例一致，可空字段、枚举、金额、时间和分页语义明确。
- 后端缺失接口有明确路径、方法、请求体、响应体和负责人，不用页面硬编码临时替代。
- 后端不可用或返回错误时，页面显示真实错误态而非模拟成功数据。
