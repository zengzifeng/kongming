## 一、本版算法修订（相对初版）

初版 Step C 用 **23:00 单点冗余** 判断集群能供出多少机器，导致「白天满载、深夜空闲」的集群（如 DeepSeek-V3.2）被误判为可供出、机器被搬空 → 按全天波形跑时峰值掉量。已做三项修订：

1. **供出口径改为峰值可行性**（`_plan_reallocation` / `_acquire_machines`）
   - `min_self_required[m] = max(0, 该模型客户波形峰值 − 三方额度)`：三方接不住、削峰也削不掉的峰值必须留自建；
   - `model_slack[m] = 该模型自建容量 − min_self_required[m]`，**只有 slack 内的自建容量可供出**；搬运逐模型递减，任何模型都不会被搬到接不住峰值的地步。取代 realtime 的 23:00 瞬时冗余（`current_redundant_*`）。
2. **新增硬约束 `model_peak_capacity_sufficient`**：成案终检，逐模型验证 `自建(调整后)+三方 ≥ 客户波形峰值`，违反即判方案不可下发。
3. **专属集群保留台数**（`_shared.py::_min_reserve_machines`）：KSCC **与 XISHANJU** 均强制保留 ≥2 台，即使当前无客户跑量也不供出。

配套数据修正：DeepSeek-V3.2 自建机器数 `8 → 10`（总能力 2600w，刚好覆盖其 2509w 峰值）；DeepSeek 确无三方额度，靠自建兜底。
